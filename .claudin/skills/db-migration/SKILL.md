---
name: db-migration
description: Generate and apply an Alembic migration after adding or changing a SQLModel table in legendarr_backend. Use when a model in modules/backend/src/legendarr_backend/**/models.py is added or changed and the database schema needs to catch up.
---

Steps to take, in order:

1. Make the SQLModel change (new table, new column, ...) in the owning slice's
   `models.py`.
2. If this is a **new model module**, import it in
   `modules/backend/db/migrations/env.py` next to the existing
   `language_profiles.models` import — Alembic's `--autogenerate` only sees models that
   are imported there.
3. Run `make db-revision message="..."` (short, imperative message describing the schema
   change, e.g. `"add translation_provider to language_profile"`).
4. Read the generated file under `modules/backend/db/migrations/versions/` and check the
   autogenerated `upgrade()`/`downgrade()` actually match the intended change — autogenerate
   can miss things like server defaults or renames (it sees them as drop+add).
5. Run `make db-upgrade` to apply it locally, then `make test` to confirm nothing broke.

Don't hand-edit `sqlalchemy.url` in `modules/backend/alembic.ini` or in `env.py` — it's left
blank on purpose so `env.py` resolves it from `Settings`/`config.yaml` for standalone CLI runs,
while `database.py::init_db()` sets it programmatically at app startup. `env.py` only fills it
in `if not config.get_main_option("sqlalchemy.url")`, so don't remove that guard.
